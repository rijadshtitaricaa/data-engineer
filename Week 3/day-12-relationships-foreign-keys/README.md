# Day 12 - Relationships, Foreign Keys, and JOINs

## Project Goal

The goal of this project was to understand how relational databases work using multiple connected tables. I practiced creating relationships between tables using primary keys and foreign keys, inserting valid and invalid data, and using SQL JOINs to create business reports.

This project focused on database structure, data integrity, and retrieving meaningful information from related tables.

## Database Tables

The database contains four main tables:

### customers
Stores customer information.

Columns:
- customer_id
- customer_name
- city

### products
Stores product information.

Columns:
- product_id
- product_name
- category
- price

### orders
Stores order information.

Columns:
- order_id
- customer_id
- order_date
- status

### order_items
Stores the products included in each order.

Columns:
- order_item_id
- order_id
- product_id
- quantity

## Primary Keys

Primary keys are unique identifiers for each table.

Used primary keys:

- customers.customer_id
- products.product_id
- orders.order_id
- order_items.order_item_id

Primary keys prevent duplicate records and allow tables to be connected safely.

## Foreign Keys

Foreign keys create relationships between tables by referencing primary keys from other tables.

Examples:

- orders.customer_id references customers.customer_id
- order_items.order_id references orders.order_id
- order_items.product_id references products.product_id

Foreign keys maintain data integrity by preventing invalid references.

## Auto Increment

Auto Increment automatically generates unique IDs when inserting new records.

For example, when inserting a customer without providing customer_id, the database creates the next available ID automatically.

Benefits:
- Prevents duplicate IDs
- Reduces manual mistakes
- Makes inserting new records easier

In real databases, IDs should normally be generated by the database instead of manually selected.

If a row is deleted, AUTO INCREMENT does not reuse the old ID. The next inserted record receives a new unique ID.

## Relationships

The database uses these relationships:

### One customer → Many orders

A customer can have multiple orders.

Relationship:
customers.customer_id → orders.customer_id

### One order → Many order items

An order can contain multiple products.

Relationship:
orders.order_id → order_items.order_id

### One product → Many order items

A product can appear in multiple orders.

Relationship:
products.product_id → order_items.product_id

These relationships allow the database to store data efficiently without duplication.

## Valid and Invalid Insert Tests

Valid inserts:

A valid insert contains existing foreign key values.

Example:
- Adding an order with an existing customer_id.

Invalid inserts:

An invalid insert contains a foreign key that does not exist.

Example:
- Adding an order with customer_id = 999 when that customer does not exist.

Foreign keys prevent invalid data from entering the database.

## INNER JOIN vs LEFT JOIN

### INNER JOIN

INNER JOIN returns only records where matching data exists in both tables.

Example:
- Showing customers that have orders.
- Showing products that were sold.

If there is no match, the record is not returned.

### LEFT JOIN

LEFT JOIN returns all records from the left table and matching records from the right table.

Example:
- Showing all customers, including customers without orders.
- Showing all products, including products that were never sold.

LEFT JOIN is useful when we want to find missing relationships.

## Business Reports

Using JOINs and aggregation functions, I created business reports such as:

- Completed revenue by city
- Completed revenue by product category
- Top 5 customers by completed revenue
- Top 5 products by completed revenue
- Customer order counts
- Items per order
- Products sold multiple times

Business rule:

Only orders with status = 'completed' are included in revenue calculations.

Pending and cancelled orders can appear in operational reports but should not affect completed revenue.
